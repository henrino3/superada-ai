Your AI agent crashes in production. Was it the LLM? Probably not. It was probably a 402 payment error your code didn't handle.

You wouldn't ship a SaaS product without tests. But add "AI" to the label and somehow testing becomes optional. The agent calls Stripe, gets rate-limited, and crashes? "Well, LLMs are unpredictable." No. Your error handling is just broken.

We built CTRL—Close The Running Loop—after six months of watching our own agents fail. Not because GPT-4 hallucinated. Because we forgot to check if an API key was set. Because Slack changed their webhook payload. Because race conditions. The boring stuff you'd catch with tests.

## The Problem With Standard Test Pyramids

The classic test pyramid (unit → integration → E2E) doesn't map cleanly to AI agents. Your agent isn't just CRUD endpoints. It's calling Stripe for payments. OpenAI for completions. Fireflies for meeting transcripts. Slack for messages.

Third-party APIs that can:
- Change their response format
- Start rate-limiting you  
- Go down entirely
- Require new auth flows

Your unit tests won't catch "Stripe added a required field to their webhook payload and now your payment processor silently fails."

## The CTRL Pyramid: Four Layers

We split testing into four layers, each targeting different failure modes:

### Layer 1: Unit Tests (< 1 second)

Pure logic. Data transformations. Validation. No API calls. No database. Just functions.

Run these on every commit.

**Example:** If your agent parses meeting transcripts to extract action items, test the parser. Mock the transcript, verify the extraction.

```javascript
test('extracts action items from transcript', () => {
  const transcript = "Alice: I'll send the deck by Friday";
  const items = extractActionItems(transcript);
  expect(items).toContain({ owner: 'Alice', task: 'send deck', due: 'Friday' });
});
```

### Layer 2: E2E Tests (1-5 minutes)

Your API routes. Auth. Database. Still mocking third-party APIs, but testing real integration between YOUR components.

Run these before every push.

**Example:** Test that a webhook endpoint validates signatures, writes to the database, and returns the right status code.

### Layer 3: Live API Tests (5-30 minutes)

Real Stripe calls. Real OpenAI calls. Real Slack messages.

These burn API credits, so don't run them on every commit. Run them pre-release or nightly.

**Why you need them:** Because Stripe WILL change their API and you want to know before your customers do.

### Layer 4: Docker Cold-Start Tests (5-10 minutes)

Spin up your entire stack from scratch. Migrations. Environment variables. Dependencies. Full production simulation.

This is your CI/CD gate. If it doesn't start clean, it doesn't ship.

## When to Use Each Layer

Not every project needs all four layers on day one:

- **Unit tests:** Always. If you write functions, test them.
- **E2E tests:** If you have API routes. Pure local scripts can skip this.
- **Live API tests:** If you integrate third-party APIs. Local-only agents can skip this.
- **Docker tests:** If you deploy to production. Laptop-only side projects can skip this.

Start with unit tests. Add layers as you mature.

## What We're Actually Testing

We're running CTRL on three projects:
- Ringmaster (RFP processing)
- Entity (workspace automation)  
- An internal CRM tool

Most runtime crashes come from:
- Null pointer errors (forgot to check if API response has expected field)
- Auth failures (token expired, wrong scope)
- Rate limiting (no backoff/retry logic)
- Database constraint violations (unique key collision)

The LLM itself? Rarely the problem. The infrastructure around the LLM is what breaks.

## Building This Into Your Workflow

We use Vitest. Setup:

```bash
npm install -D vitest @testing-library/react jsdom
```

Folder structure:
```
src/
  lib/
    parser.ts
    parser.test.ts        # unit tests next to code
  routes/
    api.ts  
    api.e2e.test.ts       # E2E tests next to routes
tests/
  live/                   # Live API tests separate
    stripe.test.ts
  docker/
    cold-start.test.ts
```

CI pipeline:
```yaml
- name: Unit tests
  run: npm test -- --run
- name: E2E tests
  run: npm run test:e2e  
- name: Docker cold-start
  run: npm run test:docker
```

Live API tests run nightly via cron to avoid burning credits on every commit.

## Automated Fix Pipeline

Testing catches bugs before production. Error monitoring catches what slips through.

Our automation:
1. Sentry webhook fires on new error
2. n8n receives it, formats context
3. Codex agent spawns with error trace + code
4. Agent writes a fix + test
5. Opens PR for review

Not fully autonomous—we review every PR. But it turns "error reported" into "fix proposed" in under 3 minutes.

This works because CTRL reduces the volume of errors. What remains are edge cases worth investigating.

## The Real Cost of No Tests

Without tests, you're doing vibe-driven development. "I ran it three times and it worked, ship it."

Then it breaks in production because:
- Someone's Slack token expired (never tested expired auth)
- Stripe webhook payload changed format (never tested parsing)
- Two requests hit the same endpoint simultaneously (never tested race conditions)

The LLM's non-determinism gives you plausible deniability. "The model hallucinated." Usually the model returned a valid response and your code failed to handle it.

CTRL doesn't test the LLM's reasoning (you can't). It tests everything around the LLM. The pipes. The error handling. The boring infrastructure that keeps your agent running.
